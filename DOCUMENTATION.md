## 1. 프로젝트 목적

이 도구는 **생성형 AI(Generative AI)** 를 이용한 에이전트 기반 모델(Agent-Based Model, ABM)로, 계산사회과학(Computational Social Science) 연구를 지원합니다.

**핵심 연구 문제**  
사회적으로 다양한 배경(나이·직업·이념·소득 등)을 가진 시민들이 동일한 뉴스를 읽고 서로 토론할 때, 각자의 태도(Attitude)가 어떻게 변화하는가?

**관찰 가능한 현상**

| 현상 | 설명 |
|---|---|
| 태도 수렴(Convergence) | 반복 상호작용으로 의견 차이가 줄어드는 현상 |
| 태도 극화(Polarization) | 이념 집단 간 격차가 오히려 심화되는 현상 |
| 에코 챔버(Echo Chamber) | 유사 성향끼리 반복 대화 시 의견이 강화되는 현상 |
| 설득 효과 | 특정 논거가 다른 성향 에이전트의 태도를 바꾸는 현상 |

실제 LLM이 각 에이전트의 페르소나·기억·현재 입장을 반영해 대화를 생성하므로, 전통적 ABM의 단순 규칙 기반 모델보다 풍부한 상호작용 시뮬레이션이 가능합니다.

## 2. 기술 구성

```
브라우저 (React + Vite)
    │
    ├── 설정 페이지 (SetupView)
    │     ├── API Key 입력 (localStorage 저장)
    │     ├── 뉴스·이슈 설정
    │     └── 에이전트 5명 구성
    │
    └── 시뮬레이션 페이지 (SimulationView)
          ├── 라운드 루프 (runRound)
          │     ├── 뉴스 읽기 단계 (reading)
          │     ├── 페어링 단계 (pairing)
          │     └── 대화 생성 단계 (discussing)
          │           └── callGemini() → Google API 직접 호출
          └── 결과 패널 (ResultsPanel) → JSON 다운로드
```

### 스택

| 구분 | 기술 |
|---|---|
| UI 프레임워크 | React 18, Vite 5 |
| 스타일링 | Tailwind CSS (CDN), CSS Variables (다크 테마) |
| 차트 | Recharts (태도 궤적 라인 차트) |
| AI 모델 | Google Gemini 2.5 Flash-Lite |
| API 통신 | 브라우저 Fetch API → Google Generative Language API (CORS 지원) |
| 상태 관리 | React useState / useRef (별도 상태 라이브러리 없음) |
| 데이터 저장 | 브라우저 메모리 (세션 내) + JSON 파일 다운로드 |

### Gemini API 직접 호출

서버 프록시 없이 브라우저에서 Google API를 직접 호출합니다.

```
POST https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash-lite:generateContent?key={API_KEY}
Content-Type: application/json

{
  "contents": [{ "role": "user", "parts": [{ "text": "<prompt>" }] }],
  "generationConfig": {
    "temperature": 0.8,
    "topP": 0.95,
    "maxOutputTokens": 2048,
    "responseMimeType": "application/json"
  }
}
```

API Key는 사용자가 직접 입력하며 `localStorage`에만 저장됩니다. 외부 서버로 전송되지 않습니다.

## 3. 에이전트 생성

### 에이전트 속성 (Agent Profile)

각 에이전트는 다음 8가지 속성으로 구성됩니다.

| 속성 | 타입 | 설명 | 시뮬레이션 내 역할 |
|---|---|---|---|
| `name` | 문자열 | 이름 | 대화 화자 식별, 로그 표시 |
| `age` | 정수 | 나이 | 프롬프트에 포함되어 LLM이 세대 특성 반영 |
| `gender` | 선택 | 성별 | 프롬프트 맥락 제공 |
| `location` | 문자열 | 거주지 | 지역 정서 반영 (수도권/지방 등) |
| `occupation` | 문자열 | 직업 | 이슈 관련 이해관계 형성 |
| `income` | 문자열 | 소득 수준 | 경제적 이해관계 반영 |
| `personality` | 문자열 | 성격·기질 | 대화 스타일 결정 (공격적/수용적 등) |
| `values` | 문자열 | 핵심 가치관·신념 | **태도 변화의 핵심 제약 조건** |
| `ideology` | 선택 | 정치 성향 | 색상 코딩 및 범주 분류용 |
| `initialAttitude` | -5 ~ +5 | 초기 입장 점수 | 시뮬레이션 출발점 |

### 이념 분류 체계

```
강경보수(-5) ─── 보수(-3) ─── 중도(0) ─── 진보(+3) ─── 강경진보(+5)
   (빨간색)        (장미색)     (보라색)     (청록색)       (진청색)
```

숫자는 기본 초기 입장 권장값이며, 이념 카테고리와 초기 입장은 독립적으로 설정 가능합니다.

### 프롬프트에 삽입되는 에이전트 정보

대화 생성 시 각 에이전트 정보는 다음 형식으로 LLM에 전달됩니다.

```
[에이전트 A]
홍길동 (45세 남성, 서울 마포구 거주)
직업: 중학교 교사 / 소득: 중산층 / 성향: 진보
성격: 논리적이고 차분함. 교육 불평등 문제에 민감.
핵심 가치관: 공교육 강화와 사회 안전망이 최우선. 시장 논리가 교육을 망친다고 생각.
현재 입장 점수: 3
최근 기억:
   1) 강 씨가 규제보다 자율이 더 효율적이라는 데이터를 제시했다.
```

## 4. 뉴스 자극 및 토론 질문 설정

### 구성 요소

| 필드 | 역할 |
|---|---|
| **헤드라인** | 뉴스 제목. 에이전트가 처음 접하는 자극. |
| **기사 내용** | 200~400자 내외의 뉴스 본문 요약. 찬반 양측 논거를 포함할수록 효과적. |
| **토론 질문** | 에이전트가 점수로 응답하는 이슈 명제. **반드시 -5~+5 척도 설명을 포함**. |

### 작성 권장사항

- 헤드라인과 내용에 **찬성 논거**와 **반대 논거**를 모두 포함하면 중도 에이전트가 흔들릴 여지가 생깁니다.
- 토론 질문은 모호하지 않게 단일 명제로 작성합니다.
- 에이전트의 직업·가치관과 연관성이 높은 이슈일수록 태도 변화가 풍부하게 나타납니다.

### 예시

```
헤드라인: 내년도 최저임금 1만 5000원 인상안 국회 제출

기사 내용: 정부·여당이 현행 9,860원인 최저임금을 내년 1월부터 1만 5,000원으로
올리는 법안을 제출했다. 지지 측은 "생존권 보장"을, 반대 측은 "중소기업 고용
감소"를 주장한다.

토론 질문: 내년도 최저임금을 1만 5000원으로 대폭 인상하는 것에 찬성하십니까?
(-5: 강력 반대 ↔ +5: 강력 찬성)
```

## 5. 시뮬레이션 1 Step 구조

하나의 **라운드(Round = Step)** 는 다음 3개 단계로 구성됩니다.

```
ROUND N
│
├── [1] 뉴스 읽기 (reading) — 0.6초 시각적 표시
│     모든 에이전트가 동일한 뉴스를 읽는 것으로 간주.
│     (실제 API 호출 없음 — 뉴스 내용은 이후 대화 프롬프트에 포함됨)
│
├── [2] 페어링 (pairing) — 0.4초 시각적 표시
│     에이전트들을 무작위로 섞어 2인 1조 쌍 구성.
│     5명 기준: 2쌍 생성 (1명은 해당 라운드 대화 없음)
│
└── [3] 대화 생성 (discussing)
      각 쌍에 대해 순차적으로 Gemini API 호출.
      5명 기준: 라운드당 2회 API 호출.
```

### ❓ 자주 묻는 질문 — 라운드당 대화 횟수

> **"5명이면 4명이 모두 짝을 이뤄 C(4,2)=6번 대화가 이루어지나요?"**

아닙니다. **라운드당 2번의 대화만 발생합니다.**

| 5명 (A B C D E) | 라운드 예시 |
|---|---|
| 셔플 결과 | [C, A, E, D, B] |
| 1번 쌍 | C ↔ A → **대화 1회** |
| 2번 쌍 | E ↔ D → **대화 1회** |
| 짝 없음 | B → 이번 라운드 대화 없음 |
| **합계** | **라운드당 2회 대화** |

- 라운드마다 매번 새로 셔플하므로, 누가 짝이 되는지·누가 쉬는지는 완전 무작위입니다.
- 모든 가능한 쌍 조합(C(5,2)=10가지)이 균일하게 나타나지는 않습니다. 같은 쌍이 연속으로 배정될 수도 있습니다.
- 5라운드 기준 총 10회 대화가 발생하며, 이 10회가 5명 사이에 무작위로 배분됩니다.

### 라운드 수와 총 API 호출 횟수

| 라운드 수 | 에이전트 수 | 쌍 수/라운드 | 총 API 호출 | 예상 소요 시간 |
|---|---|---|---|---|
| 5 | 5 | 2 | 10 | 약 1~2분 |
| 10 | 5 | 2 | 20 | 약 2~4분 |
| 5 | 4 | 2 | 10 | 약 1~2분 |
| 5 | 6 | 3 | 15 | 약 2~3분 |

> **15 RPM 제한**: 호출 간 최소 4.5초 대기가 자동 적용됩니다.  
> 503 과부하 에러 시 5초→10초→20초 간격으로 최대 3회 재시도합니다.  
> **일일 한도**: 기기(브라우저)당 50회로 제한됩니다.

## 6. 대화 상대 선택 (페어링)

페어링은 **완전 무작위 셔플** 방식을 사용합니다.

```javascript
// 피셔-예이츠(Fisher-Yates) 셔플 후 순차 쌍 구성
function pairAgents(agents) {
  const shuffled = shuffle(agents)            // 무작위 순서 재배열
  const pairs = []
  for (let i = 0; i < shuffled.length - 1; i += 2)
    pairs.push([shuffled[i], shuffled[i+1]])  // [A,B], [C,D] ...
  return pairs
  // 홀수 에이전트 → 마지막 1명은 해당 라운드 대화 없음
}
```

**특징**

- 동일한 쌍이 매 라운드 반복될 수도 있고 아닐 수도 있습니다 (완전 무작위).
- 이념적으로 유사한 쌍 또는 대립하는 쌍이 확률적으로 배정됩니다.
- 5명 중 매 라운드 1명은 대화를 하지 않아 태도가 변하지 않습니다.

## 7. 대화 생성 구조

### 프롬프트 구조

한 쌍의 대화마다 단일 Gemini API 호출로 **대화 전체 + 태도 변화 + 기억**이 한 번에 생성됩니다.

```
시스템 지시
  └── "두 시민의 대화를 시뮬레이션, 페르소나와 기억 반영"

[주제] 헤드라인 / 내용 / 질문

[에이전트 A] 이름·나이·성별·거주지·직업·소득·성향
  성격 / 핵심 가치관 / 현재 입장 / 최근 기억 (최대 2개)

[에이전트 B] (동일 구조)

[작성 규칙] (아래 참조)

[출력] JSON 형식만 반환
```

### 대화 규칙 (프롬프트 지시)

1. **턴 수**: 4~6턴 (A→B→A→B 순서)
2. **발화 길이**: 1턴당 1~2문장
3. **논거 방식**: 직업·경험에서 나온 구체적 근거. 추상적 슬로건 금지.
4. **대립 강도**: 성향 차이가 클수록 날카롭게 대립, 인신공격 없이 논거로 응수.
5. **언어**: 자연스러운 한국어 구어체

### LLM 출력 형식 (JSON)

```json
{
  "dialogue": [
    {"speaker": "A", "text": "최저임금 인상은 자영업자에겐 사형 선고예요."},
    {"speaker": "B", "text": "그건 과장이에요. 실제 데이터를 보면..."},
    {"speaker": "A", "text": "..."},
    {"speaker": "B", "text": "..."}
  ],
  "a_new_attitude": -4,
  "a_memory": "상대방이 실질임금 데이터를 근거로 들어 일부 설득됨",
  "a_reason": "구체적 통계 수치에 반박하기 어려웠음",
  "b_new_attitude": 3,
  "b_memory": "사업주의 실질적 부담을 과소평가했음을 인정함",
  "b_reason": "현장 경험 사례가 예상보다 설득력 있었음"
}
```

## 8. 태도 변화 메커니즘

### 점수 체계

```
-5 ──────── -3 ──────── 0 ──────── +3 ──────── +5
강력 반대    반대       중립       찬성       강력 찬성
```

### 변화 규칙 (LLM 지시)

| 변화 폭 | 조건 |
|---|---|
| 0 (불변) | 기본 상태. 논거가 크게 설득력 없을 때. |
| ±1 | 상대 논거가 일부 수용됐거나 새 정보를 얻었을 때. |
| ±2 | 매우 강한 설득 (드물게 발생). |
| ±3 이상 | **금지** — 핵심 가치관에 반하는 급격한 변화 차단. |

### 가치관의 역할

`values` 필드는 태도 변화의 핵심 제약 조건입니다. LLM은 이 가치관에 반하는 방향으로는 태도를 크게 변화시키지 않도록 지시받습니다. 예: 강경보수 에이전트가 단 1회 대화로 찬성(+)으로 전환되는 일은 발생하지 않아야 합니다.

## 9. 기억 시스템

각 에이전트는 대화마다 1개의 **기억(Memory)** 을 생성합니다.

- **생성 방식**: LLM이 대화 후 "새로 알게 된/느낀 점"을 40자 이내 한 문장으로 요약.
- **저장 한도**: 최대 6개 (오래된 기억 자동 삭제 — FIFO).
- **프롬프트 활용**: 다음 대화 생성 시 최근 기억 2개를 프롬프트에 포함.
- **역할**: 이전 대화의 맥락이 후속 대화에 영향을 미쳐 **태도 변화의 연속성**을 구현.

```
기억 흐름 예시:
Round 1: "보수 측 주장에서도 일부 합리적인 경제 논리가 있음을 인정"
Round 2: "노동자 실태 수치가 내 예상보다 심각했음"
Round 3 프롬프트: 위 2개 기억이 에이전트 A의 맥락에 포함됨
```

### ❓ 자주 묻는 질문 — 에이전트 간 기억 전이 여부

> **"같은 LLM이 5명으로 둔갑해 사용되는데, 다른 에이전트의 기억이 전이되지 않나요?"**

**전이되지 않습니다.** 설계상 완전히 격리되어 있습니다.

**격리 메커니즘:**

1. **LLM은 상태가 없다(Stateless)**: Gemini API는 각 호출이 독립적입니다. 이전 API 호출의 내용을 다음 호출이 자동으로 기억하지 않습니다. 프롬프트에 명시적으로 넣어야만 LLM이 알 수 있습니다.

2. **기억은 브라우저 메모리에 분리 저장**: 각 에이전트의 기억은 React 상태(state) 안의 개별 배열로 관리됩니다.
   ```javascript
   // 각 에이전트가 자신의 memories 배열을 독립적으로 보유
   agent_A.memories = ["보수 주장에서 합리성 발견", ...]   // A만의 기억
   agent_B.memories = ["진보 측 데이터가 설득력 있었음"]   // B만의 기억
   agent_C.memories = []                                    // C는 아직 대화 없음
   ```

3. **프롬프트에 해당 쌍의 기억만 포함**: A↔B 대화 프롬프트에는 A의 기억과 B의 기억만 삽입됩니다. C, D, E의 기억은 프롬프트에 포함되지 않으므로 LLM이 알 수 없습니다.

   ```
   A↔B 대화 프롬프트:
     [에이전트 A] ... A의 최근 기억 2개 ...
     [에이전트 B] ... B의 최근 기억 2개 ...
     ← C, D, E의 기억은 포함되지 않음
   ```

4. **기억 업데이트도 해당 쌍에만 적용**: 대화 후 `a_memory`는 A에만, `b_memory`는 B에만 추가됩니다. 나머지 에이전트의 기억 배열은 변경되지 않습니다.

**결론**: 동일한 LLM을 사용하지만 각 API 호출은 완전히 독립적이고, 에이전트별 기억은 코드 레벨에서 분리 관리됩니다. 기억 전이는 구조적으로 불가능합니다.

## 10. 극화 지표

### 산출 방법

라운드 종료마다 모든 에이전트 태도의 **표준편차(σ)** 를 극화 지표로 사용합니다.

```
σ = sqrt( Σ(attitude_i - mean)² / N )
```

| σ 값 | 해석 |
|---|---|
| 0.0 ~ 1.0 | 매우 낮은 극화 — 태도 수렴 경향 |
| 1.0 ~ 2.0 | 보통 수준 |
| 2.0 ~ 3.0 | 높은 극화 — 의견 분극 진행 중 |
| 3.0 이상 | 매우 높은 극화 |

**초기 기본값 예시 (최저임금 시나리오):**  
σ_초기 = `√((-5-0)²+(-3-0)²+(0-0)²+(3-0)²+(5-0)²)/5` ≈ **3.27**

```javascript
}
```

### 503 재시도

```
503 에러 발생
  → 5초 대기 후 재시도 (attempt 1)
  → 10초 대기 후 재시도 (attempt 2)
  → 20초 대기 후 재시도 (attempt 3)
  → 모두 실패 시 에러 로그 출력, 해당 쌍 건너뜀
```

### 일일 사용 한도 (기기당)

브라우저 localStorage 기반으로 기기당 하루 **50회** API 호출로 제한합니다.

```javascript
const DAILY_CALL_LIMIT = 50

// 호출 전 확인
checkDailyLimit()   // 초과 시 에러 throw → 시뮬레이션 중단
```

- 한도 도달 시 헤더에 빨간색 경고가 표시되고 이후 API 호출이 차단됩니다.
- 5라운드 5명 시뮬레이션(10회)은 하루 최대 5번 실행 가능합니다.

## 12. 사용 방법

### Step 1: API Key 준비

1. [Google AI Studio](https://aistudio.google.com) 접속 → Google 계정 로그인
2. "Get API key" → "Create API key" → 키 복사 (AIzaSy…)
3. 앱 첫 화면 API Key 입력란에 붙여넣기

### Step 2: 이슈 설정

- **헤드라인**: 뉴스 제목 입력
- **기사 내용**: 200~400자 내외 본문 요약 (찬반 양측 논거 포함 권장)
- **토론 질문**: "~에 찬성하십니까? (-5: 강력 반대 ↔ +5: 강력 찬성)" 형식

### Step 3: 에이전트 구성

- 5명 각각의 나이·성별·거주지·직업·소득·성격·가치관 설정
- **가치관 필드가 태도 변화 패턴에 가장 큰 영향을 미칩니다**
- 초기 입장 슬라이더로 -5~+5 설정
- "↺ 기본값으로 초기화" 버튼으로 예제 에이전트 복원 가능

### Step 4: 시뮬레이션 실행

- "시뮬레이션 시작 →" 클릭 → 시뮬레이션 페이지로 이동
- ROUNDS 설정 (기본 5, 최대 20)
- "▶ 시뮬레이션 시작" 클릭
- 실시간으로 대화 로그, 태도 변화 차트, 에이전트 카드 업데이트

### Step 5: 결과 저장

- 라운드 진행 중 또는 완료 후 **RESULTS 패널**에서 "↓ JSON 다운로드" 클릭
- 시뮬레이션 전체 데이터가 JSON 파일로 저장됩니다

## 13. 결과 해석

### 태도 궤적 차트 (TRAJECTORIES)

- X축: 라운드 번호
- Y축: 태도 점수 (-5 ~ +5)
- 각 선: 에이전트별 태도 변화 궤적
- **수렴**: 선들이 가운데로 모이는 경향
- **발산/극화**: 선들이 더 멀어지는 경향
- **고착**: 선이 거의 수평 (가치관이 강해 변화 없음)

### 에이전트 카드 (AGENTS)

- 현재 태도 점수 및 슬라이더 위치로 실시간 입장 확인
- `(+1)` `(-2)` 등 델타값 표시로 최근 변화량 확인
- 최근 기억 표시 — 어떤 논거에 영향받았는지 파악

### 대화 로그 (DIALOGUES)

- 라운드·페어링 정보 및 각 에이전트의 태도 변화 요약
- 실제 생성된 대화문 전체 확인
- 하단의 이유(reason) 텍스트로 태도 변화 근거 파악

### σ (표준편차) 모니터링

- 시뮬레이션 중 차트 우측 상단에 현재 σ 값이 표시됩니다
- σ 값이 라운드를 거치며 증가하면 **극화**, 감소하면 **수렴**

## 15. 한계 및 주의사항

### LLM의 본질적 한계

- **확률적 출력**: 동일한 설정으로 반복해도 다른 결과가 나옵니다. 재현 가능성(reproducibility)이 낮습니다.
- **프롬프트 의존성**: 가치관 필드를 어떻게 작성하느냐에 따라 결과가 크게 달라집니다.
- **한국어 편향**: Gemini의 한국어 이해 수준에 따라 미묘한 뉘앙스가 왜곡될 수 있습니다.
- **태도 변화 과소**: LLM이 안정적인 출력을 선호해 실제보다 태도 변화 폭이 작게 나올 수 있습니다.

### 설계상 한계

- 에이전트 수가 홀수이면 매 라운드 1명이 대화에 참여하지 않습니다.
- 페어링이 완전 무작위이므로 같은 쌍이 연속으로 매칭될 수 있습니다.
- 뉴스 "읽기" 단계는 시각적 표시만이며, 뉴스 내용은 대화 프롬프트에 직접 포함됩니다 (별도 읽기 API 호출 없음).

### 연구 활용 시 권고사항

- 동일 조건을 최소 3~5회 반복 실행 후 평균 경향을 분석하세요.
- `temperature: 0.8`은 어느 정도의 무작위성을 가집니다. 재현성이 중요하다면 `temperature: 0`을 고려하세요 (코드 수정 필요).
- JSON 결과를 Python(pandas)이나 R로 불러와 통계 분석에 활용할 수 있습니다.
